# SVG 工作流程图补充标准

本补充标准用于进一步提高 SVG 工作流程图的专业度，应与基础标准配合使用。

## 一、文档插图定位

1. SVG 应是独立的文档插图，而不是 PPT 页面。
2. 插图应能嵌入 Markdown、Notebook、网页或 PPT 中作为内容图解使用。
3. PPT 的页面标题、页脚、装饰背景由外部版式处理，不应强行画进 SVG 流程图里。
4. 不要使用“左上角大标题 + 副标题 + 下方大图”的演示页布局。
5. 如果需要标题，标题应是插图内容的一部分，短而克制；也可以完全不写标题，让外部文档标题承担说明。
6. 标题或说明文字不得挤占主体图形空间。主体结构应占据画布主要区域。
7. 不要给整张插图额外套一层装饰性背景框、蓝色外框或大圆角外壳。语义容器可以存在，但必须表达真实结构关系，不能只是为了“装起来”。
8. 画布边缘留白应克制。不要把主体缩在中间，再用四周大面积空白或背景框填充。

## 二、语义补充要求

1. 图中每个形状的角色必须清楚，例如动作、系统、工具、数据、反馈、停止条件。
2. 箭头表达的是顺序、依赖、反馈还是退出，不能只是装饰。
3. 如果任务要求表达闭环、反馈或异常路径，应让读者看清：
   - 主路径是什么；
   - 反馈从哪里产生；
   - 反馈回到哪里；
   - 什么时候退出循环；
   - 哪些节点属于主流程，哪些节点属于辅助或异常处理。
4. 选择构图时，应先服从语义，不要为了画得复杂而强行使用环形、网状或多层结构。

## 三、构图模式

先找视觉隐喻，再选择构图模式。不要默认画成卡片流程图。

一张好图应该满足两个条件：

1. 删掉大部分文字后，图形结构本身仍能让人猜出在讲什么；
2. 图中的元素关系能通过位置、箭头、层级、颜色或大小看出来，而不是只靠文字说明。

如果一张图的信息放进 Markdown 表格也能表达得差不多，就不应该画图，或者应该换一种更有视觉隐喻的构图。

根据表达对象选择合适构图：

- **线性流程**：适合表达从输入到输出的步骤，节点沿一条主轴排列，反馈或异常路径作为辅助线。
- **中心辐射**：适合表达一个核心概念与周边能力，例如核心系统与工具、规则、记录等组件。
- **容器/外壳结构**：适合表达运行环境、平台能力或模块边界，外层容器表示环境，内部核心表示主体对象。
- **左右对比**：适合表达“使用前 / 使用后”“人工流程 / 自动流程”等对比关系，中间用少量连接或 VS 标记。
- **分层结构**：适合表达平台层、工具层、执行层、验证层等上下关系。
- **卡片编组**：适合表达多个并列模块，每组卡片应有清晰标题、统一尺寸和一致间距。

更细的选择规则：

- 有明确先后顺序，选线性流程。步骤卡片必须有具体说明，不要只有标题。
- 需要反复迭代直到达标，选循环流程。循环图不加水印编号，出口路径要清楚。
- 表达“人工 / 自动”“使用前 / 使用后”，选左右对比。左右两侧结构尽量对称，中间只用轻量 VS 或连接。
- 表达成熟度、能力等级、阶段递进，选阶梯递进。用高度或颜色深浅表达“向上提升”，不要只横向排卡片。
- 表达两个维度的决策或分类，选 2×2 矩阵。十字轴和象限标签必须清楚。
- 表达核心与外围、边界与环境，选同心圆或容器结构。容器必须有语义，不能只是外框。
- 表达平台层、工具层、执行层、验证层，选分层结构。
- 如果一张图需要表达太多关系，优先拆成两张图，不要把箭头、弧线、颜色、圆点全部堆在一张图里。

关系表达手段最多使用两种。例如“箭头 + 颜色高亮”通常已经足够；不要同时使用箭头、弧线、虚线、圆点、时间线和多种颜色。

## 四、布局补充要求

1. 如果使用网状结构或多组件结构，应突出主路径，辅助路径用更浅的线条或更小的节点表示。
2. 节点应按网格或统一轨道对齐，避免随意漂浮。
3. 同一层级节点尺寸应一致或有明确层级差异。
4. 闭环图中的节点应沿清晰路径排列，箭头应沿同一方向推进。
5. 容器图应明确“内部核心”和“外部环境”的空间关系。只有当“容器”本身有语义时才画容器边界；不要把整张图外面再包一个无语义外框。
6. 不同模块之间应有足够留白。宁可减少节点，也不要让图显得拥挤。
7. 主体结构应优先占据 80% 以上的有效画布宽度或高度，避免像居中的小型 PPT 插画。
8. 如果图中同时表达“证据材料”和“结果变化”，必须分层处理。报告、截图、run log 属于证据层；分数达标、问题减少、能力增强属于结果层，不能画成同一等级的普通卡片。
9. 表达“减少”“下降”“收敛”时，视觉方向必须与语义一致，例如条形逐步变短、曲线向下或数量减少。不要用向上趋势线表达“减少”。
10. 表达“增强”“进阶”“成熟度提升”时，应使用台阶、柱形增长、等级递进或明确向上的箭头；文字标签不能压在图形或箭头上。

## 五、连线与箭头要求

1. 箭头必须从一个对象的边缘指向另一个对象的边缘，不能悬空。
2. 箭头终点应落在目标对象边缘或连接点上，不能飘在目标对象上方、旁边或空白处。
3. 箭头不能穿过节点文字。
4. 箭头不能与节点边框大面积重叠。
5. 主流程箭头应统一线宽和箭头大小。
6. 建议线宽为 1.5-2.5px，重要反馈线可略粗，但不应显得笨重。
7. 箭头头部大小应与线宽匹配，不应过大。
8. 主流程和反馈路径可以用不同颜色，但必须有明确语义：蓝色表示主流程，橙色表示反馈、风险或停止。
9. 虚线只用于反馈、可选路径或辅助关系，不用于所有主流程。
10. 多条线靠近时应保持间距，不能形成一团线。
11. 如果连线较多，应优先调整布局，而不是继续增加交叉线。
12. 动画线条应表达流程推进或反馈流动，不应让箭头显得杂乱。
13. 箭头从一个层级指向另一个层级时，终点应对准目标容器或目标区域的中心线，不能偏到旁边造成“飘移”。

闭环轨道要求：

1. 如果使用圆形或椭圆闭环，必须先定义一条标准轨道。
2. 先确定中心点 `cx, cy` 和半径 `r`，或椭圆半径 `rx, ry`。
3. 所有闭环节点的位置应由角度计算得到，不手写随意坐标。
4. 所有闭环箭头应沿同一条圆/椭圆轨道绘制。
5. 箭头终点必须落在目标节点边缘，不能落到文字区域。
6. 如果某段弧线明显偏离轨道，应重新计算轨道点，而不是靠肉眼调整。

## 六、文字排版补充要求

1. 文字应放在节点内部或节点附近，不贴边、不压线、不跨越连接线。
2. 文字与线条至少保持可见间距，线条不能从文字中穿过。
3. 若文字过长，应缩短标签，而不是压缩字距或缩小字号。
4. 节点内部文字必须水平居中和垂直居中。
5. 两行文字应作为一个整体居中，行距稳定。
6. 文字不能贴近节点边框，建议与边框至少保持 10-14px 内边距。
7. 节点高度必须能容纳文字，不允许文字挤压、溢出或被图标遮挡。
8. 不要用很小字号塞入解释性文本。解释性文本应放到正文，而不是放进流程图。
9. 同类节点用词应保持同一粒度，例如都用名词短语或都用动词短语。
10. 概念名过长时，优先用短标签，例如“检查”“记录”“停止”，不要写成长句。
11. 核心节点或中心概念的文字可以更大，建议在 22-28px；普通节点建议 16-20px；辅助说明不低于 13-14px。
12. 如果图中有中心标题或核心概念标签，应让它在视觉上成为明确焦点，不要用很小的字弱化。
13. 底部如果需要一句总结或说明，应使用深灰色或半黑色说明条承载，文字使用白色。推荐背景色 `#1F2937`、`#374151` 或带透明度的 `rgba(17, 24, 39, 0.88)`，文字不低于 14px。
14. 底部说明条只能放一句短结论，不承载长段解释；高度应克制，通常为 36-48px。
15. 图表式元素应遵守语义一致性。仪表盘、进度环、半圆弧等必须使用同一中心点和半径绘制底轨与高亮轨，不能出现底轨和高亮弧线不重合。
16. 结果标签不得压在图形、容器边框或趋势线上。标签与图形元素至少保持 8px 间距。

## 七、图标要求

1. 图标是可选增强项，不是必需项。
2. 图标必须帮助理解节点含义，不能只是装饰。
3. 节点空间不足时，优先保留文字，删除图标。
4. 图标不能与文字争抢节点中心位置。
5. 如果节点内同时放图标和文字，应明确上下或左右分区，不能重叠。
6. 图标风格应统一，优先使用 outline 风格，线宽统一。
7. 图标颜色应克制，通常使用主蓝色或灰色，不要使用多彩装饰图标。
8. 不要使用复杂人物、表情、吉祥物、贴纸式图标。
9. 不要为了让图“更丰富”强行加图标。图标造成混乱时，应删除。

## 八、风格补充要求

1. 优先使用“浅背景 + 白色卡片 + 细描边 + 轻阴影”的卡片式信息图。
2. 大面积背景应保持干净，可以用极浅蓝、极浅灰或透明背景承托主体。
3. 主要信息通过模块位置、大小、描边和少量颜色区分，不靠强装饰。
4. 可以使用小型标签、胶囊标题或分组标题来说明模块类别，但标签要短。
5. 对比图中左右两侧应有相近结构，便于读者扫描差异。
6. 中心概念图可以使用一个明确核心节点，但核心节点应是主题对象、核心系统或核心能力，不应把外部环境概念误画成普通步骤节点。
7. 卡片边角应统一，圆角适中，避免过圆导致玩具感。
8. 阴影应轻，主要用于区分层级，不用于制造漂浮装饰。
9. 蓝色用于主结构、主路径和关键标题。
10. 橙色用于警告、反馈、缺口、变化或强调项。推荐主橙色为 `#FF6A00`；需要深色文字或描边时可用 `#D95A00`；需要浅橙背景时可用 `#FFF4E8` 或 `#FFF7ED`。
11. 橙色应少量使用，通常只用于 1-3 个关键元素。
12. 不使用大面积装饰背景、氛围图、人物照片、卡通贴纸或与流程无关的视觉资产。
13. 不使用强烈营销感、发布会感或 PPT 封面感的构图。
14. 不使用过重阴影、过粗描边、过大箭头、过密渐变。
15. 优先让结构、留白、线条和层级产生美感，而不是靠装饰填充画面。
16. 图应在缩小到文档宽度时仍能读清主要节点。
17. 如果使用设备窗口、浏览器框或文件夹结构，应让它服务信息结构，不要只是装饰外壳。
18. 不使用无语义的整图外框。整张插图的边界应由内容自然形成，而不是靠外层边框定义。

## 九、创意质量要求

1. 图形结构必须服务概念，不要把文字塞进方框后就算完成。
2. 每张图只能有一个主要视觉隐喻，例如阶梯、闭环、三角关系、同心边界、左右对比、漏斗或矩阵。
3. 同一章节中的多张图不应全部使用同一种卡片流程。应根据内容选择不同构图。
4. 图中节点必须使用具体概念，不使用“步骤一”“模块 A”“能力 B”这类抽象占位符。
5. 关系必须可视化。如果两个元素有因果、递进、反馈或对立关系，必须用位置、箭头、层级或颜色表达。
6. 图里文字只放短标签和关键结论。正文已经讲清的解释，不要搬进图里。
7. 底部结论条只写一句总结，不写口号，不写完整段落。
8. 如果图看起来像表格，先判断是否真的需要画图。能用表格等效表达的内容，不应强行变成插图。
9. 一张图的关系表达手段不超过两种。超过两种时，优先删除装饰性最强、语义最弱的那一种。
10. 生成前必须先说明选择该构图的理由。理由应基于内容结构，而不是“看起来更丰富”。

## 十、通用插图质量补充规则

以下规则用于提升文档插图的通用质量，不绑定某个具体实验、Skill 或工具。生成和验收时，应重点检查语义、层级、文字、图形与连线是否一致。

### 10.1 先判断语义，再决定图形结构

1. 不要在没有判断环节的情况下，直接把“未达标”“异常”“问题”连到某个修订对象。应先判断问题来源，再决定是修订产物、修订规则、补充标准，还是增强验证能力。
2. 如果图中表达“判断来源”“分流处理”“继续/停止”等逻辑，必须画出判断节点或分流节点，不能用一条线直接跳过判断。
3. 回路必须表达清楚“修订后回到哪里”。如果是重新生成，就回到生成节点；如果是补规则或补标准，也应再回到生成节点或执行节点，而不是停在修订节点。
4. 图形结构必须服务概念。不要为了画得复杂而强行使用环形、网状、多层嵌套或多条回路线。

### 10.2 区分证据层、过程层和结果层

1. 截图、报告、日志、审查意见、测试结果等属于证据层。
2. 生成、验证、判断、修订、停止等属于过程层。
3. 分数达标、问题减少、能力增强、质量提升、进入生产等属于结果层。
4. 不同层级不应全部画成同款卡片。证据层可以用卡片，过程层可以用流程或闸门，结果层应使用仪表、趋势、阶梯、徽章或其他结果型视觉。
5. 结果层的视觉语义必须和文字一致：
   - “减少”“下降”“收敛”应用递减条、向下趋势或数量变少表达；
   - “增强”“提升”“成熟”应用阶梯、柱形增长、等级递进或向上箭头表达；
   - “达标阈值”应用同心同轨的仪表、刻度或明确阈值线表达。

### 10.3 不自创无来源的成熟度模型

1. 如果没有明确引用来源，不要自创 L0-L4、L1-L5 等“成熟度模型”。
2. 表达从实验到生产、从试点到推广、从个人到团队时，优先使用左右对比、能力补齐图、路线图或治理清单。
3. 如果需要表达阶段位置，应明确说明“已具备哪些能力”和“还缺哪些能力”，不要含糊标注“处于某某入口”。
4. 生产化类图应强调目标标准、专家参与、长期指标、权限预算、版本治理、人审门禁等工程要素。

### 10.4 概念表达要准确，不要把上层概念画成普通节点

1. 如果概念表示“外部运行环境”“治理层”“控制面”“平台能力”，应使用容器、外壳、分层或控制台结构，而不是画成普通步骤节点。
2. 如果概念表示“循环机制”“多轮收敛”“反馈推进”，应突出状态推进、验证、写回和停止条件，而不是只画一个圆形箭头。
3. 如果两个概念存在包含关系，不要画成并列关系。例如“运行环境提供约束，循环机制在其中运转”就不应画成两个并列卡片。
4. 图中所有概念标签都应能回答“它代表什么”。不能用“模块 A”“能力 B”“步骤一”这类占位词。

### 10.5 箭头、标签和背景的硬性检查

1. 所有 SVG 必须有显式白色背景，避免透明背景在 Notebook、Markdown 或 PPT 中显示异常。
2. 箭头终点必须落在目标对象边缘或目标区域中心线，不能偏到旁边或飘在空白处。
3. 关键标注文字不能压在弧线、趋势线、边框或箭头上。发现遮挡时，优先移动标注或改用标签容器。
4. 回路线如果需要绕回起点，应优先使用外侧直角线或清晰弧线，不能从证据块、结果块或底部说明条中穿过。
5. 底部说明条应与主图保持明显间距。反馈线、回路线和趋势线不能贴住或压住底部说明条。
6. 如果使用圆/椭圆闭环或仪表盘，底轨和高亮轨必须来自同一组几何参数。不要凭肉眼手写近似弧线。
7. 多条连线靠近时，应优先调整布局，而不是继续增加弯曲线、虚线或装饰线。

### 10.6 文字风格和图注

1. 图注应写成工程上可理解的短句，例如“未达标项补全标准，规则按新标准执行”。不要写含糊口号。
2. 图内文字要短，只放概念标签和关键结论。正文已经讲清的解释，不要搬进图里。
3. 避免使用过度负面的表达。优先使用“未达标项”“问题项”“验证反馈”“不通过项”“待修订项”等中性工程词。
4. 标题、节点和说明条的字号应形成层级：标题最大，节点次之，说明文字最小但仍可读。
5. 同类标签应保持同一语法粒度。不要一边用名词短语，一边用完整句子。
6. 中英文混排时要核算宽度。英文缩写、产品名或术语不要挤压中文标签。

### 10.7 验收时必须做三类检查

1. **语义检查**：删掉一半文字后，图形结构是否仍能表达核心关系？
2. **视觉检查**：是否存在文字压线、箭头悬空、弧线跑偏、标签遮挡、层级混乱、留白异常？
3. **用途检查**：这张图是否适合作为文档插图嵌入，而不是像 PPT 封面、海报或营销页？
